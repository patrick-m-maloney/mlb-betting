  Current state

  The data is organized across three layers plus raw:

  data/
  ├── raw/              # Timestamped Parquet snapshots (odds, schedules, player logs)
  │   ├── odds/YYYY-MM-DD/games/   # ~688K

  Assessment

  What's working well:
  - Timestamped Parquet snapshots for odds and lineups (append-only, never overwrite) — this is the right approach for
  time-series betting data where you need line movement history.
  - Hive-style partitioning on silver lineups (year=/month=/day=) is good for DuckDB reads.
  - Reference data cleanly separated.

  Problems:
  1. DuckDB is severely underutilized — only 18 rows across 2 tables. The append_to_table function works, but almost no
  data flows through it. Most data is in Parquet files that DuckDB can't easily query without manual view creation.
  2. Dual-write pattern (Parquet files + DuckDB inserts) means data can diverge. Some fetchers write to Parquet only
  (odds snapshots via save_snapshot()), some to DuckDB only (via append_to_table()), some to both.
  3. No single query layer — to answer "what were the odds for today's games?", you'd need to glob Parquet files
  manually. DuckDB should serve as the query layer over all Parquet files, but views are only created for lineups.
  4. season_totals/ at 40MB is the largest directory but uses misleading filenames and isn't integrated with any
  downstream consumer.
  5. No partitioning on odds data beyond date folders. During the regular season with polling every 10-30 minutes, each
  day could produce 50+ small Parquet files in data/raw/odds/YYYY-MM-DD/games/ — that's fine for now but will slow down
  reads.

  Recommended target architecture

  Principle: Parquet as storage, DuckDB as the query layer.

  Keep writing timestamped Parquet snapshots (they're your immutable audit trail). But register ALL of them as DuckDB
  views so everything is queryable from one connection:

  data/
  ├── bronze/                    # Raw ingestion (append-only Parquet)
  │   ├── lineups/YYYY-MM-DD/   # (already here)
  │   ├── odds/YYYY-MM-DD/      # Move from data/raw/odds/

  │   └── polymarket/YYYY-MM-DD/
  ├── silver/                    # Cleaned, deduped, enriched
  │   ├── lineups/year=*/month=*/day=*/  # (already here)
  │   ├── odds/year=*/month=*/day=*/     # Flattened odds per game/book/market
  │   └── player_stats/                  # Merged batting + pitching + wOBA
  ├── gold/                      # Model-ready features
  │   ├── projections/           # Per-player projected wOBA/FIP for today
  │   └── edges/                 # Model probability vs market odds
  ├── reference/                 # (keep as-is)
  ├── simulations/               # (keep as-is)
  └── db/mlb_betting.duckdb      # Views over ALL parquet; tables for small data

  1. Consolidate odds into bronze/odds/ instead of raw/odds/. One storage tier per data type.
  2. Create DuckDB views at startup (in db_manager.py) for every Parquet directory — bronze_lineups, bronze_odds,

  3. Stop dual-writing — write Parquet first (audit trail), then let DuckDB views read it. Remove direct INSERT INTO for

  - The timestamped append-only Parquet pattern is correct and should be kept.
  - The Hive partitioning on silver lineups is good.

  - The comps_cache.parquet for KNN historical data is fine.
